一 · 雲端基礎設施與 CI/CD
1. ECS → EKS 全面遷移 里程碑完成

這個月最大的一件事。生產前端原本跑在 ECS（AWS 容器服務），五月把它整批搬到自管的 EKS（Kubernetes）叢集。切換不是一次到位，而是透過 Cloudflare Load Balancer 的權重把使用者流量分批導向新叢集——先導一小部分、觀察穩定、再加碼，全程使用者無感（zero-downtime）。

為什麼要搬：EKS 讓我們能用 Karpenter 自動調度節點、混用 spot 機器省成本、把 CI runner / imgproxy / cloudflared 等周邊都收進同一個可觀測、可 GitOps 管理的平台。這條主線也帶動了下方第 2~3 項（ARC、可觀測性）的所有工作。

  • trends-stg EKS 遷移（5/22 / 5/25）：staging 已切 EKS、prod 仍 EC2（kubectl live verify 確認），切換後移除 envoy BasicAuth 簡化。

  • spot posture 決策（5/29-30）：100% spot 不符 zero-downtime（PDB 擋不住 spot 回收），改為 on-demand baseline + spot burst（70/30，HPA 4-44）。
  • 其他：sosreader-client EKS 遷移、preview 環境 K8s 遷移。

2. ARC on EKS — self-hosted CI runner phase 6 達成

原本 CI（GitHub Actions）跑在 GitHub 託管的機器上、按分鐘計費。ARC（Actions Runner Controller）讓我們把 runner 跑在自己的 EKS 上，build 快取與映像層可控、長期攤提成本。五月把它從 phase 0 推到 phase 6，核心難點都在 buildkitd（負責 container build 的元件）的權限與快取。
